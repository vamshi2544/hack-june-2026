THE PROBLEM

Today, the release lifecycle has two ends and a gap in the middle.

Engineers and release leads work hard before the deploy. On-call works hard if something breaks after. The window in between — the four-to-twenty-four hours where the new code is alive and probationary — is nobody's full-time job.

▸ ReleasePilot · pre-deploy

Reads Jira release stories, builds the Confluence release page, fills the Excel template, advances the ServiceNow CHG through approvals, and — critically — assembles a risk packet of past INCs and known failure modes for this application.

▸ ReleaseWatch · post-deploy

Wakes when the deploy completes. Watches metrics across Splunk, New Relic, the app and its upstream/downstream dependencies. Compares baselines, reasons about drift, escalates proportionally, and — when something is wrong — diagnoses with bitbucket evidence.

PHASE 1 · 30 SECONDS

ReleasePilot, briefly. What happens before the deploy.

A human comments @release-pilot go on a closed Jira story. The agent wakes and works the entire release in parallel.

Read & understand

Reads the Fix Version, finds every release story under it, reads all subtasks, classifies what's in this release.

Document & advance

Builds the Confluence release page. Fills the Excel template. Attaches both to the CHG. Advances state through approval gates as humans confirm in Teams.

Assess risk

Pulls every past INC on this application from ServiceNow. Identifies common failure modes. Writes a risk packet for ReleaseWatch to read after deploy.

The risk packet is the architectural insight. Pilot does the homework; Watch reads it and knows exactly what to look for. The two agents feel like one coordinated system.

PHASE 2 BEGINS

ReleaseWatch wakes the moment production deploy completes.

Two equally-supported trigger paths. The agent doesn't care which fires — same outcome.

▸ ServiceNow comment trigger

The deployment lead (or the deploy pipeline) adds a comment to the CHG: Deployed to CDE2 or Deployed to CDE3. ReleaseWatch reads this on its poll, picks up the CHG, loads the risk packet, starts watching.

▸ Pipeline webhook trigger

UDeploy posts a webhook to ReleaseWatch with {chg, foundation, app}. Deterministic, no parsing ambiguity. The production path.

Foundation-by-foundation discipline. SYF's rollout is staged — CDE2 at 1am, then if clean, CDE3 the next day. ReleaseWatch watches each foundation independently. Its CDE2 findings become the go/no-go signal for CDE3 progression. A clean CDE2 watch is a credential, not just an absence of alerts.

THE ESCALATION MODEL

Three levels. Cadence increases as severity rises.

The agent's job isn't to alert. It's to communicate proportionally — quiet when quiet is appropriate, urgent when it isn't.

▸ L1 · Watching
All clear
Default state. Compares current metrics to last week same time, yesterday same time, pre-deploy baseline. Notes expected drift (from risk packet). All within range.
Hourly all-clear email
▸ L2 · Amber
Something is off
Pattern detected. Agent reasons about cause. Considers release-notes-expected behavior. If unexplained, surfaces to humans.
Immediate email + 15-min updates
▸ L3 · Red
Issue is real
Pattern persists or worsens. Agent searches past INCs, pulls bitbucket diff, generates RCA. Creates ServiceNow INC with consolidated evidence. Posts Adaptive Card to Teams. Every minute matters now.
INC + Teams + per-minute updates

Human override is built-in. Once at L3, an on-call engineer can acknowledge the Teams card and downgrade to L2 — the INC stays open for audit, but the cadence relaxes. Human is now driving. If a related symptom re-appears later, the agent first checks for an active INC and appends to it rather than double-paging.

LIVE DEMO · THE SCENARIO

A real failure mode from our team's experience.

Tonight's deploy: acquisition-decision-orchestrator. A producer app that sends messages over RabbitMQ to a downstream consumer for async decisioning.

▸ 1:00 AM CDE3

Deploy completes successfully. All health checks green. ReleaseWatch reads the CHG comment, loads the risk packet, starts watching the producer and its downstream consumer.

▸ The silent failure

The deploy looked clean. But a subtle config change means producer messages get queued slower. Over hours, RabbitMQ memory fills. Backpressure begins. The producer logs nothing. The consumer logs nothing. Both apps look healthy individually. Async decisions just start arriving late.

This is the killer scenario. No error logs. No alerts. No single system showing red. The only signal is: "the consumer hasn't processed anything new in 20 minutes." Today, on-call discovers this hours later when downstream consequences accumulate. ReleaseWatch catches it in 8 minutes — because it watches the relationship, not just the systems.

INSIDE THE AGENT · L2

How ReleaseWatch goes from "metric drift" to "this isn't noise."

▸ ReleaseWatch · L2 reasoning trace · 02:08 AM
Comparing current window to baselines for CHG0049182.
Producer (acquisition-decision-orchestrator): message-send rate normal · CPU normal · memory normal · no error logs.
Consumer (decision-async-consumer): message-receive rate down 78% vs same time yesterday · message-receive rate down 81% vs same time last week · decision-completion latency up 340%.

Producer looks healthy in isolation. Consumer looks degraded in isolation. The relationship — producer "claims" to be sending, consumer "claims" to be waiting — that's the anomaly. Both apps are part of the same risk packet. Cross-app correlation is exactly what the risk packet anticipated. Checking release notes: enrich_v3 toggle expects +15ms latency on producer. Consumer-side latency expansion of 340% is not explained by release notes.

No INC active for either app in the last 4 hours. Symptoms are unexplained by intended release behavior. This is amber. Escalate. Action: Set CHG0049182 → AMBER. Email DL dl-acquisition-oncall@syf.com. Begin 15-min update cadence.

The reasoning that matters: producer and consumer each look fine alone. The agent is watching the relationship, which is something that exists in the risk packet but not in any single dashboard. That's the cross-system view today's monitoring can't deliver.

INSIDE THE AGENT · L3 · THE RCA

From "something is wrong" to "here is the cause, with confidence."

L2 has run 30 minutes. Pattern is worsening. Agent escalates to L3 and begins diagnosis.

▸ STEP 1 · Past INC search · ServiceNow
Query: similar symptom pattern on this application family in the last 180 days.

Match found: INC0218763 · 2026-04-12 · "decision-async-consumer · message backlog · root cause: rabbitmq node memory_high_watermark exceeded, backpressure to producers, no application-level logs."

Match found: INC0223440 · 2026-05-29 · same root cause, different application family.

Pattern similarity: HIGH
▸ STEP 2 · Pull live RabbitMQ metrics · New Relic
Querying RabbitMQ cluster rmq-acquisition-prod-cde3...

Node 1: memory 92% · HIGH WATERMARK EXCEEDED
Node 2: memory 88% · approaching watermark
Backpressure flag: active · started 01:47 AM
Producer connections: throttled

Live evidence: confirms hypothesis from past INCs
▸ STEP 3 · Bitbucket diff · last deploy
Pulled diff: acquisition-decision-orchestrator · commits since last green deploy.

5 commits. One stands out:
+ messageProps.setPersistent(true) in DecisionMessagePublisher.java

Persistent messages take ~3x longer to enqueue and consume more memory than transient. At normal volume this is fine. At post-deploy traffic spike volume, it pushed the cluster past the high watermark. The change is correct for durability but missed a capacity-planning check.
▸ STEP 4 · Consolidated RCA
Root cause: Commit a7f3c19 enabled persistent RabbitMQ messages in DecisionMessagePublisher. Persistent messaging tripled memory pressure on the cluster. Within 47 minutes of deploy, memory crossed the high watermark and backpressure was applied. Producer was throttled, consumer starved.

Immediate fix: Either (a) increase RabbitMQ cluster memory · or (b) revert the persistent-message flag pending capacity planning. Option (b) is faster.

Long-term: Add cluster-capacity check to release readiness for any change to message-durability flags.

DEFENDING THE PITCH

"Isn't this just better alerting?"

The reasonable critique. Here's why it doesn't apply.

▸ Better Alerting (rules engine)
  • "If rabbitmq.memory > 85% then page on-call."
  • Each alert is independent. A producer alert and a consumer alert are two pages, not one diagnosis.
  • No way to reason about cross-app relationships ("producer fine but consumer starved").
  • No way to reason about release intent ("approval rate dropped because release notes said it would").
  • Cannot search past INCs by symptom similarity — only by exact keyword.
  • Cannot read code diffs. Cannot reason about whether a code change explains a symptom.
▸ Agent (Copilot Studio reasoning)
  • Watches relationships the risk packet declared, not just thresholds.
  • Reasons about whether observed drift is explained by release intent in the Confluence notes — and lowers severity if so.
  • Searches past INCs by symptom similarity, not exact match. Finds related failures even with different keywords.
  • Reads bitbucket diff. Reasons about whether the change plausibly causes the observed behavior.
  • Consolidates multiple evidence streams into a single coherent RCA — with confidence.
  • Generalizes. The same agent works for App A and App B; the watch profile differs but the reasoning doesn't.

Alerts tell you something happened. The agent tells you what, why, and how confident it is. That's not a rule. That's judgment over context — exactly what humans do in the first 30 minutes of an incident, and exactly what they can be relieved of.
