# 06 · SharePoint Release Page Template

> The Confluence-styled release page that ReleasePilot generates. This is the most visible artifact in the demo besides the Adaptive Card.

## What this page contains

The release page is the human-readable artifact for a given release. It carries the risk packet (which ReleaseWatch reads later) plus all the human-friendly context (scope, deployment plan, approvers).

```
┌────────────────────────────────────────────────────────┐
│ Release Page · 2026.06.08 · Acquisition Decision Orch. │
├────────────────────────────────────────────────────────┤
│ ▸ Header (banner)                                      │
│ ▸ Scope and Intent                                     │
│ ▸ Deployment Plan                                      │
│ ▸ Change List (from Jira Fix Version)                  │
│ ▸ Risk Packet (auto-generated · 4 subsections)         │
│ ▸ Approvers                                            │
│ ▸ Notification Preferences                             │
│ ▸ Footer (auto-stamped)                                │
└────────────────────────────────────────────────────────┘
```

## Building the template in SharePoint

4. Page URL slug: `Release-2026-06-08`

> This section is auto-generated by ReleasePilot and is read by ReleaseWatch

| INC0214098 | 2026-02-22 | Thread pool exhaustion in enrichment | Low — different code path |
| INC0218763 | 2026-04-12 | RabbitMQ memory_high_watermark exceeded | **HIGH** — this release affects message persistence which directly impacts memory usage |
| INC0223440 | 2026-05-29 | Same root cause as INC0218763 on sibling app | **HIGH** — same failure mode, different application |

### Known failure modes

- `rabbitmq_backpressure` — producer throttled when broker memory exceeds high watermark
- `thread_pool_exhaustion` — enrichment thread pool saturated under high concurrent load
- `downstream_consumer_starvation` — consumer falls behind when producer rate exceeds processing rate
- `null_pointer_in_enrichment` — historical bug pattern in enrichment path

| Application | Role | Why watch |
|-------------|------|-----------|

- **Consumer throughput: NO change expected.**

## How the agent uses this page

During the demo:
1. **ReleasePilot** sees the Jira trigger, gathers context, then calls SharePoint **Create Page** action with this template + the synthesized risk packet text
2. The page is created in the AI Pulse Releases site
3. Page link is added to CHG Tickets list (the demo CHG row)
4. Notification email and Teams message include the page link
5. **ReleaseWatch** reads the risk packet section of this page after deploy — that's the handoff

## Pre-deploy state vs post-deploy state

The page is mostly static. ReleaseWatch only updates:
- The status banner (Pre-deploy → Watching → Recovered or INC-Open)
- The footer (last-updated timestamp + state)

The risk packet section never changes. It's the immutable contract.

## Common issues

| Symptom | Fix |
|---|---|
| Markdown not rendering | SharePoint Text web parts support markdown only in some tenant versions · use the Rich text formatting bar if markdown doesn't render |
| Tables look broken | Use the Table web part instead of markdown tables |
| Studio can't update the page | Verify Studio agent has Edit permission on the site |
| Page edits show concurrency errors | SharePoint caches page state · wait a few seconds between edits |
