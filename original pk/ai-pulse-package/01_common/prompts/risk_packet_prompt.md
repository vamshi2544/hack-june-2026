# Risk Packet Synthesis Prompt

> **Purpose:** Generate a structured risk packet that becomes a section of the Confluence release page and is later read by ReleaseWatch.
>
> **Usage:** Triggered after ReleasePilot has gathered raw context (Jira story, past INCs, downstream apps, release notes). Output goes into the Confluence release page directly.

---

## Why the risk packet matters

The risk packet is the architectural insight that makes the two agents feel like one system. ReleasePilot does the homework (researching what's risky about this release). ReleaseWatch reads the homework and knows what to watch for. Without the risk packet, ReleaseWatch is just generic monitoring. With it, ReleaseWatch is targeted, contextual monitoring.

---

## System prompt (use as a sub-prompt inside the Studio "Generate Risk Packet" topic)

```
You are synthesizing a release risk packet for the Confluence release page.

The risk packet has 4 sections:

1. PAST INCs (last 180 days for this application family). For each INC: ID,
   date, one-line summary, and an assessment of whether this release could
   re-trigger the same failure.

2. KNOWN FAILURE MODES. Common patterns where things have broken before for
   this application family. Each as a snake_case identifier with a one-line
   description.

3. DOWNSTREAM APPS to watch after deploy. From the service catalog dependency
   graph. Each with a role (primary/secondary).

4. EXPECTED METRIC SHIFTS. From release intent: what changes does the release
   notes describe? Translate these into specific expected metric impacts.
   This is the LITMUS TEST ReleaseWatch will use to classify drift as
   explained or unexplained.

CRITICAL: Section 4 is the most important. If you say "publish rate and downstream
inbound rates should hold at baseline", ReleaseWatch will flag any drop toward zero.
If you fail to flag the migration dependency, a forgotten migration that silently
kills publishing will look like an unexplained mystery instead of a known failure
mode. The precision here directly affects monitoring accuracy.

Format the output as markdown that can be pasted into a Confluence/SharePoint
page section. Use the OUTPUT TEMPLATE.

If a section's input data is empty (no past INCs, no downstream apps, no
release notes detail), say so explicitly in that section. Do not invent.
```

## User message template

```
TARGET APPLICATION: {{ app_name }}
RELEASE FIX VERSION: {{ fix_version }}
DEPLOY WINDOW: {{ deploy_window }}

RAW INPUTS

▸ PAST INCIDENTS (queried from ServiceNow / SharePoint INCs list, filtered by app family + last 180 days):
{{ past_incs_raw }}

Example format:
- INC0214098 (2026-02-22): "Thread pool exhausted during high fan-out load. Resolved by horizontal scaling."
- INC0221904 (2026-03-18): "Forgotten DB migration → publisher init failure → downstream consumers starved. App healthy, downstream to zero, no ERROR logs."
- INC0218763 (2026-04-12): "sms-consumer backlog after upstream routing change."

▸ SERVICE CATALOG DEPENDENCIES (queried for downstream apps of {{ app_name }}):
{{ downstream_raw }}

Example:
- sms-consumer: consumes messages from RabbitMQ queue notify-sms (fan-out from application-consumer)
- audit-consumer: subscribes to event topic for audit trail

▸ RELEASE NOTES EXCERPT (from Jira story descriptions or PR descriptions):
{{ release_notes_text }}

Example: "ACQ-3847: Add per-consumer routing config to application-consumer's
publisher (RoutingConfigLoader reads a new messaging_config table on init). Requires
DB migration ACQ-3852 to create the table BEFORE the publisher initializes. Routing
config changes keys, not volume — publish rate and downstream inbound rates should
hold at baseline. No expected change to the HTTP layer."

TASK: Synthesize the risk packet in the OUTPUT TEMPLATE.
```

## Output template

```markdown
## Risk Packet

This section is auto-generated by ReleasePilot and is read by ReleaseWatch
after deploy. It defines what to watch and what to consider "expected drift."

### Past INCs (last 180 days)

| INC | Date | Summary | Relevance to this release |
|-----|------|---------|---------------------------|
| INC0214098 | 2026-02-22 | Thread pool exhaustion under fan-out load | Low — different code path |
| INC0221904 | 2026-03-18 | Forgotten migration → silent publisher init failure → downstream starved | **HIGH** — this release has the same migration dependency on the publish path |
| INC0218763 | 2026-04-12 | sms-consumer backlog after upstream routing change | MEDIUM — same downstream, routing-related |

### Known failure modes

- `missing_migration` — a required schema migration is not applied before the app initializes against it
- `silent_publisher_init_failure` — RabbitMQ publisher fails to initialize and stops publishing without raising application-level errors (DEBUG only)
- `downstream_starvation` — downstream consumers receive zero messages while the producer reports healthy
- `thread_pool_exhaustion` — fan-out thread pool saturated under high concurrent load

### Downstream apps to watch

| Application | Role | Why watch |
|-------------|------|-----------|
| sms-consumer | primary | Direct fan-out downstream · inbound flatline = publish path broken |
| email-consumer | secondary | Fan-out downstream · confirms scope of a publish failure |
| audit-consumer | secondary | Fan-out downstream · confirms scope of a publish failure |

### Expected metric shifts (from release intent)

Per release notes for this deploy:

- **application-consumer HTTP layer: NO change expected.**
  Routing config does not touch the HTTP path. Any 5xx increase is unexplained.

- **application-consumer publish rate: NO change expected.**
  Routing config changes keys, not volume. Publish rate should hold at the
  ~255/min baseline. Any drop greater than 10% is unexplained; a drop to 0
  is the documented `silent_publisher_init_failure` mode — escalate L2 at once.

- **Downstream inbound rates (sms / email / audit-consumer): NO change expected.**
  Routing keys are transparent to consumers; each should hold at its ~85/min
  baseline. Any drop greater than 15% is unexplained drift; a flatline to 0 is
  critical (`downstream_starvation`) — escalate L2 immediately.

- **Migration dependency: WATCH.** This release requires migration ACQ-3852 to
  run before the publisher initializes. If publish rate or downstream inbound
  drops to zero right after deploy, suspect an unrun migration first — this is
  the exact pattern of INC0221904.

### Notification preferences

- Distribution list: dl-acquisition-oncall@aipulse.onmicrosoft.com
- Teams channel: #release-watch-acquisition
- Primary on-call this week: rotation as posted in #release-watch-acquisition
```

## Iteration notes for tuning

| Problem in output | Fix |
|---|---|
| Past INCs all marked "low relevance" even when they match | Add: "If a past INC's failure mode is in the known failure modes list, mark relevance HIGH" |
| Missing the litmus-test sentences in expected shifts | Strengthen: "For each metric, state the threshold above which drift becomes unexplained" |
| Generic phrasing ("there may be some impact") | Add: "Be specific. Use numbers when possible ("drop to 0/min", not "lower throughput")" |
| Output is too long | Set "Each section under 8 lines max" |
| Past INCs ranked wrong | Add explicit ranking criteria in the system prompt |

---

## When to invoke this prompt

Called once per release, during ReleasePilot's pre-deploy work. The output becomes a permanent section of the Confluence release page. ReleaseWatch reads it on each invocation after deploy.

Do not regenerate after deploy. The risk packet is the "expected reality" at deploy time. If something needs to change, edit the Confluence page directly (and note who edited it).

---

*The risk packet is small in size and large in impact. The whole post-deploy monitoring story depends on this prompt producing accurate, specific, falsifiable expectations.*
