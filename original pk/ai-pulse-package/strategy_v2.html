Five paths to ship ReleasePilot + ReleaseWatch

One workflow, five ways to build it. Each path makes different trade-offs between architectural credibility, demo reliability, visual fidelity, and the network reality of a SYF laptop. The decision is which one to build for Monday — and which one to keep as a hot fallback.

01 The five paths at a glance

Read these like a menu. Each path is a complete answer to "how do we ship this." The deep treatment of each follows below.

▸ Path A
Real Production
Bedrock + Apigee + real APIs. Not buildable in 2 days — this is the pitch slide that shows the architecture story.
▸ Path B
Real Free Connections
Real Jira trigger via free webhook → real Studio LLM → real SharePoint/Teams/Outlook. Confluence/SNOW/Bitbucket simulated on SharePoint.
▸ Path C
Studio + Mock Data
Studio is the brain. SharePoint is the world. No Jira/Bitbucket integration. Cleaner than B, loses the real Jira event moment.
▸ Path D
Local App + Minimal LLM
Pretty local React app for all UIs. One Studio screen-share moment at L3 RCA to prove the LLM exists. Otherwise scripted.
▸ Path E
Pure Mock + Architecture Story
Beautiful local app. Zero LLM. Production architecture diagram at the end. Honest about being a simulation.
The asymmetry that shapes every path: Atlassian (Jira/Confluence/Bitbucket) can push events into M365 for free via webhooks, emails, web requests. M365 cannot pull from Atlassian without Power Automate Premium. M365 can talk to itself (Teams ↔ SharePoint ↔ Outlook ↔ Studio) freely. This means: any path that includes a "real" trigger from Atlassian works on free tooling. Any path that needs the agent to query Atlassian back does not.

03 Path B · Real free connections (the most interesting path)

PATH BReal Jira trigger · real Studio · real Teams/Outlook · SharePoint simulates the rest

Use only free tooling. Real Atlassian pushes real events into M365 via webhooks. Studio reasons. M365 surfaces (Teams, Outlook, SharePoint) carry out the agent's actions for real. The agent never has to "reach into" Atlassian — which is exactly the free tier's constraint, and it's fine because the demo is one-shot from Jira to closure.

▸ The Path B workflow · end to end · all free

1Vamshi closes release story in real Jira (his personal Atlassian Cloud Free).
2Vamshi comments @release-pilot go on the story.
3Jira Automation rule fires (free). Action: Send web request POST to Teams Incoming Webhook URL with payload containing story details.
4Real Teams channel receives the message with story details.
5Power Automate flow (free, "When message posted in channel" trigger) fires. Parses story details from the message.
6Flow calls Copilot Studio agent with the story context (Studio agent published to same Teams channel). Studio reasons over the context using its built-in LLM.
7Studio reads pre-seeded SharePoint list of past INCs · synthesizes risk packet using LLM. Writes risk packet as a real SharePoint page styled as a Confluence release page.
8Studio posts real Adaptive Card to Teams channel asking for approval. Human (Vamshi or teammate) clicks Approve.
9Studio updates real SharePoint list (representing the SNOW CHG) · advances state from Assess to Implement.
10Studio sends real Outlook email to test DL with CHG notification.
11Studio posts status back to Teams channel: "CHG ready for deploy. Posting back to original Jira issue would happen in prod via API."
12Deploy trigger: Bitbucket webhook (free) on PR merge OR manual trigger via SharePoint list update OR user message in Teams.
13Studio (ReleaseWatch agent) starts L1 hourly reports. Posts to Teams every hour with "all clear" or escalation.
14L2 drift: Studio reads pre-seeded SharePoint list of "current metrics" vs "yesterday metrics" · reasons over delta · classifies as explained/unexplained · posts amber alert.
15L3 RCA: Studio reads pre-seeded SharePoint lists for past INCs, RabbitMQ metrics, Bitbucket diff · synthesizes RCA with confidence score · creates SharePoint list row representing INC · posts real Teams Adaptive Card with RCA.
16Human acknowledges in Teams. Studio adjusts cadence, watches for recovery, posts all-clear when metrics improve.

Pros
  • Real Jira event wakes the agent. The most credible moment in any demo.
  • Real LLM reasoning visible in Teams thread — not narrated, not faked.
  • Real human-in-the-loop via real Adaptive Cards. Approval gates land.
  • Real emails go out — judges see the inbox notification land.
  • Architectural credibility very high — only thing missing vs prod is the API gateway.
  • Builds the discipline of pushing data into M365 — which is a transferable pattern.
Cons
  • Setup-heavy: Jira account, automation rule, Teams channel, SharePoint lists, Studio agent — all to wire up Monday morning.
  • Visual fidelity: SharePoint pages don't look like Confluence pages out of the box. Need CSS injection or accept a less polished look.
  • Many moving pieces — any one of which can break in the live demo.
  • Bitbucket integration is partial — webhook in works, but the diff data has to be pre-seeded for Studio to "read."
  • Depends on Studio credits availability on Monday.
When you use this path: if Monday's checks pass on Studio access + Teams connector + reasonable credits AND your team has the appetite to wire ~6 free integrations in an afternoon. This is the demo that wins on architectural credibility.

04 Path C · Studio + Mock Data (no Jira integration)

PATH CStudio reasons over SharePoint-stored mock data · all triggers in Teams

A simpler version of Path B. No Jira webhook, no Atlassian wiring. The demo starts when you chat with the Studio agent in Teams. The agent does everything Path B does, but the trigger is a Teams message instead of a Jira comment. SharePoint is the world.

Pros
  • Fewer moving pieces than Path B — no Jira/Bitbucket wiring.
  • Real LLM, real Teams cards, real emails.
  • Faster to build Monday morning.
  • More reliable than Path B because fewer external dependencies.
Cons
  • Loses the "real Jira event woke the agent" moment — the most viscerally credible part.
  • Pitch has to lean harder on "imagine this came from Jira" narrative.
  • Same SharePoint visual fidelity issue as Path B.
When you use this path: if Monday checks pass on Studio but Path B's wiring complexity feels risky. Or if you want to maximize Studio reasoning time and minimize setup time.

05 Path D · Local App + Minimal LLM

PATH DPretty local React app + Studio screen-share at the L3 moment

Build the vibe-coded local app (Jira / Confluence / SNOW / Teams / Dashboard) for visual fidelity. Run it on localhost. At the L3 RCA moment, switch tabs to Studio (a separate browser tab) and run the actual RCA prompt live to prove the LLM exists. No real connection — just two surfaces shown in sequence.

ComponentPath D implementation
LLMREAL Copilot Studio · shown via screen-share for L3 RCA only
TriggerSCRIPTED Type comment in mock Jira UI
All UIsMOCKED Vibe-coded React, polished visuals
All actionsSCRIPTED State machine in React
Reasoning streamsSCRIPTED Pre-written, stream into UI
L3 RCA generationREAL Switch to Studio tab · run prompt live · narrate
Connection between app and StudioNONE Demonstrator narrates the bridge
Pros
  • Highest visual fidelity. Pretty Jira, pretty Confluence, pretty SNOW, pretty Dashboard.
  • Demo flow is exactly Mock 1 you already approved.
  • One real LLM moment is enough to answer "where's the AI?"
  • Local app has zero network dependencies. Reliable.
Cons
  • The bridge between app and Studio is narration, not connection. Sharper judges will notice.
  • Architecturally weaker than B or C — looks more like a demo than a system.
  • Switching tabs mid-demo can break the visual flow.
When you use this path: if Monday checks pass on Studio basic LLM but NOT on Teams/SharePoint connectors. You can still prove the AI exists without wiring real connections.

06 Path E · Pure Mock + Architecture Story

PATH EBeautiful local app · zero LLM · honest architecture diagram at the end

Build the local app. Run the demo entirely from scripted state. At the end of the pitch, show the production architecture diagram and explain "this is where AI plugs in. Today, we don't have access to that in our hackathon environment. In prod, this becomes Bedrock or Studio reasoning." The honest, maximally-reliable choice.

ComponentPath E implementation
LLMNONE Explained as future plug-in point
All UIsMOCKED Vibe-coded React, polished visuals
All actionsSCRIPTED State machine in React
"Reasoning"SCRIPTED Pre-written text streams that look like LLM output
Architecture storyREAL Pitch slide 14 shows where AI plugs in for prod
Pros
  • Maximum reliability. Zero external dependencies. Demo works in any room.
  • Fastest to build — no Studio setup at all.
  • Demo visual quality is as high as you can make it.
  • Honesty is disarming — judges respect "we know this is mocked; here's where AI fits."
Cons
  • "Where's the AI?" is the hardest question, and the answer is "in the next slide."
  • Weakest path on architectural credibility for an "agentic AI hackathon."
  • Risks being scored down for not using the LLM tools the hackathon explicitly provides.
When you use this path: as the floor. Build Path E first. Layer Path B, C, or D on top if Monday's checks pass. If everything else fails, you ship Path E and it still works.

08 Recommendation

▸ The play
Build Path E as the floor over the weekend (throwaway prep) so the team learns the mocks. Monday morning, run the 6 checks. If they pass, layer Path B on top. If they partially pass, fall back to Path C. If they fail entirely, ship Path E and use Slide 14 to tell the architecture story honestly.
Why this play: Path B is the most impressive thing buildable, but you can't commit to it before knowing Monday's tenant capabilities. Path E is always achievable. The two are not competitors — they're floor and ceiling. Path E's polished mocks become the visual layer Path B doesn't have. Path B's real Studio reasoning becomes the AI proof Path E lacks. Together, with a feature flag, they cover every Monday outcome.

What to build this weekend

  1. Throwaway local React app covering all 5 mock UIs at Mock 1 fidelity. Vamshi leads, Anisha + Sai Mohit shadow.
  2. Pre-seeded data files — past INCs, metrics, bitbucket diff content, story details — drafted in JSON ready to drop into either React state (Path E) or SharePoint lists (Path B).
  3. The L3 RCA master prompt — drafted now, tested in Claude/ChatGPT this weekend, ready to paste into Studio Monday.
  4. The Monday-morning playbook — hour-by-hour build plan with team allocation.
  5. The Jira automation rule template — ready to paste into Jira Cloud on Monday morning to wire Path B's trigger fast.
  6. The SharePoint list schemas — column definitions for the 4-5 lists Path B needs.
Everything above is preparation, not building. Per hackathon rules, the actual code, Studio configuration, and Power Automate flows must be created after the official start time. The prep work gives you the muscle memory; Monday is when you actually ship.
